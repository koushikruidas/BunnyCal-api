Booking Cancellation & External Synchronization Architecture
Production-Grade Lifecycle Management Design

1. Overview
BunnyCal supports multiple booking lifecycle mutation paths:
* authenticated host cancellation
* anonymous guest cancellation
* external provider synchronization
* reconciliation repair workflows
* future workflow automation
* policy enforcement actions
* automated lifecycle transitions
Although lifecycle mutations originate from different actors and systems, all mutations converge into a single authoritative domain boundary:
BookingAggregate
This architecture is intentionally designed around:
* aggregate ownership
* transactional consistency
* asynchronous orchestration
* eventual consistency
* replay-safe workflows
* deterministic reconciliation
* operational resilience
* horizontal scalability

2. Architectural Goals
The system must provide:
* unified lifecycle semantics
* deterministic aggregate transitions
* replay-safe distributed workflows
* reliable asynchronous orchestration
* provider reconciliation safety
* scalable background processing
* operational observability
* auditability
* secure anonymous guest management
* future extensibility

3. Non-Goals
The system intentionally does NOT guarantee:
* distributed ACID transactions
* exactly-once delivery
* globally ordered events
* immediate provider convergence
* perfect webhook reliability
* provider-side causal ordering
* real-time synchronization guarantees
External calendar providers are treated as:
eventually consistent distributed systems

4. Core Architectural Principles

4.1 Booking Aggregate Is the Sole Lifecycle Authority
ALL lifecycle mutations MUST execute through:

BookingAggregate

No external system may directly mutate lifecycle state.
This includes:
* controllers
* workers
* reconciliation jobs
* webhook handlers
* provider adapters
* repair pipelines
Valid mutations include:

cancelByHost()
cancelByGuest()
cancelExternally()
reschedule()
markNoShow()

This guarantees:
* invariant enforcement
* concurrency correctness
* deterministic transitions
* replay safety
* centralized lifecycle semantics

4.2 Separation of Responsibilities
Aggregate Layer
Responsible for:
* lifecycle invariants
* legal state transitions
* aggregate versioning
* domain event generation

Application Service Layer
Responsible for:
* authorization
* orchestration
* aggregate loading
* transaction management
* command coordination
Example:

BookingCancellationApplicationService


Workflow / Integration Layer
Responsible for:
* provider synchronization
* notifications
* retries
* reconciliation
* analytics
* async orchestration

6. Aggregate State Machine
The lifecycle state machine MUST be explicit and deterministic.

6.1 Supported States

PENDING
CONFIRMED
CANCELLED_BY_HOST
CANCELLED_BY_GUEST
CANCELLED_EXTERNALLY
RESCHEDULED
NO_SHOW


6.2 Legal State Transitions
Current State	Command	Result
PENDING	confirm	CONFIRMED
CONFIRMED	cancelByHost	CANCELLED_BY_HOST
CONFIRMED	cancelByGuest	CANCELLED_BY_GUEST
CONFIRMED	cancelExternally	CANCELLED_EXTERNALLY
CONFIRMED	reschedule	RESCHEDULED
CONFIRMED	markNoShow	NO_SHOW
RESCHEDULED	cancelByHost	CANCELLED_BY_HOST
RESCHEDULED	cancelByGuest	CANCELLED_BY_GUEST
CANCELLED_*	any cancellation	NO-OP
NO_SHOW	cancellation	REJECTED
6.3 Invalid Transition Behavior
Invalid transitions MUST:
* return deterministic domain errors
* emit NO domain events
* trigger NO side effects
Example:

NO_SHOW -> cancelByGuest()

Result:

BookingAlreadyClosedException


7. Aggregate Concurrency Model

7.1 Optimistic Concurrency
All aggregate mutations use optimistic locking.
Example:

UPDATE bookings
SET version = version + 1
WHERE id = ?
AND version = ?


7.2 Aggregate Sequence
Every successful lifecycle mutation increments:

aggregate_sequence

The sequence is monotonic per aggregate.
Purpose:
* stale event rejection
* replay protection
* workflow ordering
* reconciliation safety

7.3 Stale Workflow Rejection
Workers MUST reject stale operations.
Rule:

incoming_sequence < current_aggregate_sequence
→ reject operation

Example:
Event	Sequence
cancel booking	10
delayed reschedule retry	9
Result:

retry ignored as stale


8. Transaction Boundaries

8.1 Strong Consistency Zone
The following operations execute atomically:
1. load aggregate
2. validate invariants
3. mutate aggregate
4. increment version
5. increment aggregate_sequence
6. persist domain event
7. insert outbox event
8. commit transaction
These operations MUST occur in one database transaction.

8.2 Eventual Consistency Zone
The following operations are asynchronous:
* provider synchronization
* email delivery
* ICS delivery
* analytics
* reminder invalidation
* webhook reconciliation

9. Transaction Isolation Requirements
The system assumes:

READ COMMITTED

with optimistic concurrency enforcement.
The architecture intentionally avoids:

SERIALIZABLE

due to scalability costs.
Correctness relies on:
* version checks
* aggregate_sequence checks
* idempotent workflows
NOT database-level serialization.

13. Idempotency Strategy
ALL commands and async workflows MUST be idempotent.

13.1 Command Idempotency
Duplicate cancellation requests MUST:
* return success
* emit NO duplicate events
* trigger NO duplicate side effects

13.2 Notification Deduplication
Notifications MUST use semantic deduplication keys.
Example:

(notification_type, booking_id, aggregate_sequence)

This prevents:
* duplicate emails
* duplicate reminders
* duplicate ICS invalidation

13.3 Webhook Deduplication
Webhook ingestion MUST enforce uniqueness:

(provider, provider_event_id)
UNIQUE

Duplicate webhook deliveries MUST become no-ops.

16. External Provider Synchronization

16.1 Synchronization Philosophy
Provider synchronization is treated as:
probabilistic reconciliation
NOT immediate consistency.

16.2 Webhook Handler Rules
Webhook handlers MAY:
* validate signatures
* validate replay windows
* enqueue sync jobs
* persist reconciliation tasks
Webhook handlers MUST NOT:
* mutate aggregates directly
* execute heavy reconciliation
* send notifications inline

17. Reconciliation Conflict Resolution
Reconciliation MUST follow deterministic precedence rules.

17.1 Conflict Matrix
Scenario	Winner
Internal cancellation vs stale provider state	Internal
Duplicate provider webhook	Ignore duplicate
Delayed provider retry	Reject as stale
Provider deletion after confirmation	External cancellation
Internal reschedule supersedes old provider state	Internal
Missing provider event	Repair sync workflow
17.2 Stale Reconciliation Rejection
Workers MUST compare:

incoming_sequence
vs
current_aggregate_sequence

Older operations MUST be rejected.

REQUIRED ENGINEERING TASKS
Before implementation begins, create:

1. Sequence Diagrams
For:
* host cancellation,
* guest cancellation,
* provider cancellation,
* retry replay,
* stale event rejection.
This helps junior engineers enormously.

2. Aggregate Test Matrix
You need exhaustive tests for:
Scenario	Expected
duplicate cancel	no-op
stale reschedule retry	rejected
concurrent host+guest cancel	deterministic winner
provider replay	ignored
already cancelled booking	idempotent success
This is critical.

3. Idempotency Contract Tests
Mandatory.
Especially:
* webhook dedupe,
* notification dedupe,
* outbox replay,
* retry replay.

4. Failure Injection Testing
You should deliberately test:
* provider timeouts,
* duplicate webhooks,
* worker crashes,
* outbox retry storms,
* DB rollback scenarios.

5. Operational Runbooks
Create:
* DLQ recovery procedures,
* replay procedures,
* provider outage handling,
* reconciliation repair guides.
This becomes important surprisingly early.
